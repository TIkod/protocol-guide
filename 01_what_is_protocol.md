# Что такое протокол (TICODE - https://ticode.online/)

## Определение

**Протокол** — это набор правил и соглашений, определяющих, как именно два (или более) участника обмениваются данными.

Аналогия из жизни: когда вы звоните незнакомому человеку, вы оба знаете протокол — сначала говорите "Алло", потом представляетесь, ждёте ответа. Если один нарушит правила и начнёт сразу говорить о делах без приветствия — возникнет непонимание. Протоколы в IT работают так же.

Формально протокол определяет:
- **Синтаксис** — структуру и формат сообщений (как выглядит сообщение)
- **Семантику** — смысл каждого поля и команды (что означает сообщение)
- **Тайминг** — порядок и время обмена (когда и в каком порядке отправлять)

---

## Зачем нужны протоколы

Без протоколов компьютеры не могут общаться — каждый производитель делал бы своё. Протоколы решают:

1. **Совместимость** — Python-сервер может отвечать на запросы JavaScript-клиента, потому что оба говорят на HTTP
2. **Надёжность** — TCP гарантирует доставку каждого пакета
3. **Безопасность** — TLS шифрует данные так, что посередник не может их прочитать
4. **Масштабируемость** — стандарт позволяет строить инфраструктуру из независимых компонентов

## Ключевые характеристики протокола

### Stateful vs Stateless (с состоянием / без состояния)

**Stateless** — каждый запрос независим, сервер не помнит предыдущих.

```python
# HTTP/1.1 — stateless
# Каждый запрос полностью самодостаточен
import httpx

# Первый запрос — сервер не помнит его при втором
response1 = httpx.get("https://api.example.com/users/1")

# Второй запрос — как будто первого не было
response2 = httpx.get("https://api.example.com/users/2")
```

**Stateful** — сервер хранит состояние соединения между запросами.

```python
# TCP — stateful
# Сервер знает, что соединение установлено, ждёт следующих пакетов
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(("example.com", 80))  # Состояние: ESTABLISHED

sock.send(b"GET / HTTP/1.1\r\nHost: example.com\r\n\r\n")
data = sock.recv(1024)  # Сервер знает, что мы — тот же клиент

sock.close()  # Состояние: CLOSED
```

### Синхронный vs Асинхронный

**Синхронный** — отправил запрос, жди ответа.

```python
import requests

# Блокирует выполнение до получения ответа
response = requests.get("https://api.example.com/data")
print(response.json())  # Код дойдёт сюда только после ответа
```

**Асинхронный** — отправил, занимайся другим, ответ придёт потом.

```python
import asyncio
import aiohttp

async def fetch_data():
    async with aiohttp.ClientSession() as session:
        # Не блокирует — можно делать другие дела пока ждём
        async with session.get("https://api.example.com/data") as response:
            return await response.json()

asyncio.run(fetch_data())
```

### Push vs Pull

**Pull (запрос-ответ)** — клиент сам запрашивает данные когда нужно.
```
Клиент → "Дай мне данные" → Сервер
Клиент ← "Вот данные"    ← Сервер
```

**Push** — сервер сам отправляет данные когда они появляются.
```
Сервер → "Появились новые данные!" → Клиент
Сервер → "Ещё обновление!"         → Клиент
```

### Binary vs Text протоколы

**Текстовый протокол** — данные передаются как читаемый текст.

Плюсы: легко отлаживать, читать в Wireshark/curl
Минусы: больший размер, медленнее парсить

```
# HTTP запрос — чистый текст
GET /api/users HTTP/1.1
Host: example.com
Accept: application/json
```

**Бинарный протокол** — данные передаются в байтах.

Плюсы: компактность, скорость парсинга
Минусы: нечитаем без инструментов

```python
import struct

# Бинарное сообщение: тип (1 байт) + длина (4 байта) + данные
message_type = 1
data = b"Hello"
packet = struct.pack(">BI", message_type, len(data)) + data
# Результат: b'\x01\x00\x00\x00\x05Hello'
```

## Анатомия протокольного сообщения

Почти любое протокольное сообщение состоит из:

```
┌──────────────────────────────────────────┐
│  HEADER (заголовок)                      │
│  - Тип сообщения                         │
│  - Версия протокола                      │
│  - Длина тела                            │
│  - Метаданные (откуда, куда, когда)      │
├──────────────────────────────────────────┤
│  BODY / PAYLOAD (тело / полезная нагрузка│
│  - Реальные данные                       │
├──────────────────────────────────────────┤
│  FOOTER / CHECKSUM (опционально)         │
│  - Контрольная сумма для проверки        │
└──────────────────────────────────────────┘
```

Пример разбора HTTP ответа:

```
HTTP/1.1 200 OK                    ← Статусная строка (часть заголовка)
Content-Type: application/json     ← Заголовок
Content-Length: 27                 ← Заголовок
                                   ← Пустая строка — граница header/body
{"name": "Alice", "age": 30}      ← Тело (body)
```

## Виды протоколов — обзор

```
Протоколы
├── Сетевые (транспорт данных)
│   ├── IP — адресация и маршрутизация
│   ├── TCP — надёжная доставка
│   ├── UDP — быстрая доставка без гарантий
│   └── ICMP — служебные сообщения сети
│
├── Прикладные (конкретные задачи)
│   ├── HTTP/HTTPS — веб
│   ├── FTP — файлы
│   ├── SMTP/IMAP/POP3 — email
│   ├── DNS — имена → IP адреса
│   └── SSH — безопасное управление
│
├── Безопасности
│   ├── TLS/SSL — шифрование канала
│   ├── OAuth 2.0 — авторизация
│   └── JWT — токены идентификации
│
├── Обмена сообщениями
│   ├── WebSocket — двусторонний реалтайм
│   ├── MQTT — IoT устройства
│   └── AMQP — брокеры сообщений
│
├── Сериализации (форматы данных)
│   ├── JSON — текстовый, универсальный
│   ├── XML — текстовый, строгий
│   ├── Protobuf — бинарный, быстрый
│   └── MessagePack — бинарный JSON
│
└── RPC (вызов удалённых функций)
    ├── REST — HTTP + ресурсы
    ├── gRPC — Protobuf + HTTP/2
    ├── GraphQL — гибкие запросы
    └── JSON-RPC — простой RPC
```

## Как протоколы договариваются (Handshake)

**Handshake** — процедура установки соединения и согласования параметров.

```
Клиент                    Сервер
  │                         │
  │──── SYN ───────────────►│   "Хочу соединиться"
  │                         │
  │◄─── SYN-ACK ────────────│   "Принял, подтверждаю"
  │                         │
  │──── ACK ───────────────►│   "Принял подтверждение"
  │                         │
  │     [соединение есть]   │
  │                         │
  │──── данные ────────────►│
  │◄─── данные ─────────────│
```

Это TCP three-way handshake. Каждый протокол имеет свой вариант.

## Порты — адреса протоколов

IP-адрес указывает на компьютер, **порт** указывает на конкретный протокол/сервис.

| Порт | Протокол | Назначение |
|------|----------|-----------|
| 80   | HTTP     | Веб (незащищённый) |
| 443  | HTTPS    | Веб (защищённый) |
| 22   | SSH      | Удалённый доступ |
| 21   | FTP      | Передача файлов |
| 25   | SMTP     | Отправка email |
| 53   | DNS      | Разрешение имён |
| 5432 | PostgreSQL | База данных |
| 6379 | Redis    | Кэш/хранилище |

```python
import socket

# Подключение к конкретному порту = выбор протокола
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(("example.com", 443))  # 443 = HTTPS
```

## RFC — стандарты протоколов

Большинство сетевых протоколов описаны в документах **RFC (Request for Comments)**, публикуемых организацией IETF.

- RFC 791 — IP
- RFC 793 — TCP
- RFC 768 — UDP
- RFC 9110 — HTTP/1.1
- RFC 8446 — TLS 1.3

Это открытые документы — любой может их прочитать и реализовать совместимый протокол.
